# Madoka Launch Engine - 要件定義書

## 1. プロダクト概要

### 1.1 プロダクト名
Madoka Launch Engine

### 1.2 コンセプト
コンテンツ販売における**セールスファネル構築からローンチ運用までを一気通貫で支援する**AIマーケティングプラットフォーム。

LP・配信文・セミナースライド・リードマグネット等のコンテンツをプロのマーケティング視点で一括生成し、ローンチ中はダッシュボードで数値管理・AI診断・追加施策提案までを行う「ローンチ司令塔」。

### 1.3 ミッション
> 「売れる」を目的とした全コンテンツを、修正がほぼ不要なクオリティで一括出力し、ローンチ全期間を通じて売上最大化を支援する。

### 1.4 利用者
- **主要ユーザー**: 円香（Madoka）本人
- **利用形態**:
  - 自社プロモーション
  - プロモーターとしてクライアント案件のローンチ構築を受託
- **操作者**: 円香のみ（シングルオペレーター）

### 1.5 売上目標
- 1回のローンチあたり最低 **10,000,000円**

## 2. プロジェクトタイプ

アプリ内で作成するプロジェクトは以下の4タイプに分類される。タイプに応じて有効化されるモジュールが変わる。

| タイプ | 規模 | 説明 | 有効モジュール |
|--------|------|------|--------------|
| フルローンチ | 高単価・大型 | 大型ファネル全体を構築 | 全モジュール |
| スポット販売 | 低〜中単価 | ハウスリストに配信して低単価商品を販売 | プロジェクト管理, 配信生成, 簡易LP, ダッシュボード(軽量) |
| リスト教育 | 売らない | 価値提供・関係構築のための配信 | プロジェクト管理, 配信生成 |
| エバーグリーン | 常時稼働 | 常時稼働するファネル | LP, ステップ配信, ダッシュボード(月次) |

### 2.1 タイプ別ファネルテンプレート

#### フルローンチ用
- **A**: オプトイン → ステップ配信 → セミナー → 個別相談 → 成約
- **B**: オプトイン → ステップ配信 → セールスLP → 成約
- **C**: VSL → セールスLP → 成約
- **D**: SNS → オプトイン → ローンチ配信(プリプリ→プリ→ローンチ) → セールス → 成約
- **E**: カスタム（ステップを自由に組み合わせ）

#### スポット販売用
- **S1**: ハウスリスト配信 → セールスLP → 成約
- **S2**: ハウスリスト配信 → 申込フォーム → 成約

#### エバーグリーン用
- **EG1**: 広告/SNS → オプトインLP → ステップ配信 → セールスLP → 成約

## 3. モジュール構成（13モジュール）

### Phase MVP（初期リリース）

#### 3.1 M01: プロジェクト管理モジュール

**目的**: 商品・ペルソナ・オファーの情報を一元管理し、全モジュールの生成基盤とする。

##### 3.1.1 商品情報入力
| 項目 | 必須 | 説明 |
|------|------|------|
| 商品名 | ○ | 販売する商品/サービスの名称 |
| 商品概要 | ○ | 何を提供するか |
| 価格帯 | ○ | フロントエンド/バックエンド/単発 |
| ターゲット | ○ | 誰向けか（概要） |
| USP | ○ | 独自の強み・差別化ポイント |
| 実績・証拠 | ○ | 数字で示せる成果、社会的証明 |
| ステータス | ○ | 新規商品 / 既存商品（過去ローンチ有） |
| 過去ローンチデータ | △ | 既存商品の場合、過去のCVR/売上等 |

##### 3.1.2 ペルソナ設計（AI壁打ち形式）

AIが対話形式で質問し、円香が答えていくことでペルソナが完成する仕組み。

| 項目 | 必須 | 用途 |
|------|------|------|
| 基本属性（年齢/性別/職業/年収） | ○ | ターゲティング |
| 悩み・課題TOP5 | ○ | コピーの問題提起パート |
| 理想の未来（ビフォーアフター） | ○ | ベネフィット訴求 |
| 今まで試した解決策と失敗理由 | ○ | 共感パート・差別化 |
| 購買時の不安・反論 | ○ | FAQ・保証設計 |
| 情報収集チャネル | ○ | SNS戦略 |
| 使う言葉・口癖 | △ | コピーのトーン調整 |

##### 3.1.3 オファー設計
| 項目 | 説明 |
|------|------|
| 価格設定 | 通常価格/早期割引/分割 |
| 特典設計 | メイン特典 + ボーナス特典（価値換算付き） |
| 保証設計 | 返金保証/成果保証/サポート保証 |
| 限定性 | 人数/期間/価格の限定要素 |
| 緊急性 | カウントダウン/早期特典/段階値上げ |

##### 3.1.4 トンマナ・世界観設定
| 項目 | 説明 |
|------|------|
| ブランドカラー | メイン/サブ/アクセント |
| フォント指定 | 見出し/本文 |
| トーン | 権威的/親しみやすい/情熱的 等 |
| 世界観キーワード | 高級感/カジュアル/プロフェッショナル 等 |

##### 3.1.5 クライアント管理
| 項目 | 説明 |
|------|------|
| クライアント名 | 自社 or クライアント名 |
| プロジェクト一覧 | クライアントに紐づく全プロジェクト |
| 実績の分離管理 | 自社/クライアント別で実績を管理 |

#### 3.2 M02: ファネル設計 & KPIモジュール

**目的**: セールスファネルの構成を設計し、売上目標からKPIを逆算、配信カレンダーを自動生成する。

##### 3.2.1 ファネル設計
- プロジェクトタイプに応じたテンプレート選択
- カスタムファネル: ステップをドラッグ&ドロップで自由構成
- ファネル選択に応じて必要なモジュール/出力物を自動有効化

##### 3.2.2 KPI逆算エンジン

```
入力:
  売上目標: ¥10,000,000
  商品単価: ¥500,000
  ファネル: オプトイン → セミナー → 個別相談 → 成約

出力:
  必要成約数: 20件
  必要個別相談数: 40件（成約率50%想定）
  必要セミナー着席数: 200件（相談移行率20%想定）
  必要セミナー申込数: 286件（着席率70%想定）
  必要オプトイン数: 572件（セミナー申込率50%想定）
  必要LP流入数: 2,860件（オプトイン率20%想定）
```

- 各ステップのCVRはデフォルト値（業界平均）をプリセット
- 過去実績があれば上書き可能
- ボトルネック診断: 各ステップのCVRを比較し改善優先度を自動判定

##### 3.2.3 配信カレンダー自動生成
- ローンチ日（セールス開始日）を入力 → 逆算で全配信日程を自動配置
- ガントチャート表示
- スポット販売は3〜7日間のシンプルカレンダー
- UTAGE設定用チェックリスト付き（「この配信をUTAGEに設定した？」）

##### 3.3.2 コピースコアリング

生成したコピーをAIが自動採点:

| 評価項目 | 配点 |
|---------|------|
| ヘッドライン（ベネフィット・好奇心） | /10 |
| 問題提起（具体性・共感度） | /10 |
| 論理構成（PASONA/QUEST等の適合度） | /10 |
| 社会的証明（実績の量と質） | /10 |
| オファー（価値の伝わりやすさ） | /10 |
| CTA（明確性・緊急性） | /10 |
| 反論処理（不安への対応） | /10 |
| 読みやすさ（文の長さ・リズム） | /10 |
| ターゲット適合度（ペルソナとの一致） | /10 |
| 独自性（テンプレ感のなさ） | /10 |

- 80点以上: そのまま使用可能
- 60〜79点: 該当箇所の改善案を自動提示
- 60点未満: 再生成を推奨

##### 3.3.3 マーケティングフレームワーク内蔵

AIが状況に応じて最適なフレームワークを自動選択:

| フレームワーク | 用途 |
|--------------|------|
| PASONA（新PASONA） | セールスLP全般 |
| QUEST | 問題認識が浅いターゲット |
| PAS | 短めのLP（オプトイン等） |
| AIDA | シンプルなオファー |
| GDTの法則 | ヘッドライン生成 |
| OATHフォーミュラ | ターゲットの認知レベル判定→トーン調整 |

#### 3.4 M04: 配信コンテンツ生成モジュール

**目的**: ファネルの各ステップに応じた配信文（LINE/メール）を一括生成する。

##### 3.4.1 配信タイプと出力

| 配信タイプ | タイミング | 通数目安 | チャネル |
|-----------|-----------|---------|---------|
| オプトイン後ステップ | 登録直後〜セミナー前 | 5〜10通 | LINE/メール |
| セミナーリマインド | 3日前/前日/当日/1時間前 | 4通 | LINE/メール |
| セミナー後フォロー | セミナー翌日〜 | 3〜5通 | LINE/メール |
| セールス期間 | セールス開始〜終了 | 5〜10通 | LINE/メール |
| カウントダウン | 締切3日前〜当日 | 3〜4通 | LINE/メール |
| 購入後オンボーディング | 購入直後〜 | 3〜5通 | LINE/メール |
| 未購入者リカバリー | セールス終了後 | 2〜3通 | LINE/メール |
| プリプリローンチ | ローンチ2〜4週前 | 3〜5通 | LINE/メール |
| プリローンチ | ローンチ1〜2週前 | 3〜5通 | LINE/メール |

##### 3.4.2 配信設計原則

| 原則 | 適用 |
|------|------|
| ザイオンス効果 | 適切な接触頻度の設計 |
| 一貫性の原理 | オプトイン〜セールスまでメッセージの一貫性 |
| 社会的証明の段階配置 | ステップ配信で徐々に実績を見せる |
| 損失回避バイアス | カウントダウン/限定性配信 |
| ストーリーテリング | 配信全体を通じたストーリー構築 |

##### 3.4.3 LINE/メール出し分け

同じ配信内容でもチャネル特性に合わせて最適化:

| チャネル | 特徴 | 最適化 |
|---------|------|--------|
| LINE | 短文、即読、開封率高 | 短めのメッセージ、リッチメニュー誘導 |
| メール | 長文可、件名重要 | 件名最適化、HTML構成、プレヘッダー |

#### 3.5 M05: ナレッジベースモジュール

**目的**: 過去の素材・ベンチマーク・成功パターンを蓄積し、AIの生成品質を継続的に向上させる。

##### 3.5.1 ナレッジ登録

| 種別 | 登録方法 | 用途 |
|------|---------|------|
| 過去LP | テキスト貼付 or URL | スワイプファイルとして参照 |
| 過去配信文 | テキスト貼付 | トーン学習、成功パターン抽出 |
| ベンチマーク | URL or テキスト | 競合分析、差別化 |
| セミナー資料 | テキスト/PDF | 構成パターン参照 |

##### 3.5.2 ベンチマーク分析（AI自動）

| 分析項目 | 出力 |
|---------|------|
| LP構成分析 | セクション構成の分解 |
| コピーパターン抽出 | ヘッドライン/CTA/オファーの型を特定 |
| 差別化ポイント提案 | 真似すべき点 / 差別化すべき点 |
| スワイプファイル自動分類 | LP/配信/SNS別に自動整理 |

##### 3.5.3 RAG検索

- ベクトルDB（Embedding）で類似素材を検索
- コンテンツ生成時に関連するナレッジを自動参照
- 円香の文体・表現パターンを学習し生成に反映

#### 3.6 M06: お客様の声・実績管理モジュール

**目的**: 社会的証明の素材を一元管理し、コピー生成時に自動引用する。

##### 3.6.1 実績データベース

| 項目 | 説明 |
|------|------|
| 成果テキスト | 「3ヶ月で月収100万円達成」等 |
| ビフォーアフター | 変化の具体的な数字/状況 |
| スクリーンショット | 成果画像（管理用） |
| 動画URL | お客様の声動画 |
| タグ | 商品別/ジャンル別/成果タイプ別 |
| 使用許可 | 公開可/匿名のみ |

##### 3.6.2 自動引用

- LP生成時、該当商品/ジャンルの実績を自動でセクションに挿入
- 実績の引用数が不足している場合は警告表示
- クライアント案件と自社案件の実績を分離管理

#### 3.7 M07: ローンチダッシュボードモジュール

**目的**: ローンチ/スポット販売中の数値をトラッキングし、AIが分析・施策提案を行う。

##### 3.7.1 数値入力

| 入力方式 | 説明 |
|---------|------|
| 日次クイック入力 | 各ステップの当日数値を入力（1分で完了） |
| CSV一括取込 | UTAGEからエクスポートしたCSV |
| 前日値プリセット | 累計値は差分入力でOK |
| スマホ対応 | レスポンシブUI |

##### 3.7.2 ダッシュボード表示

| 表示項目 | 説明 |
|---------|------|
| ファネル進捗 | 各ステップの目標 vs 実績 vs 達成率 |
| ステップCVR | 各ステップ間のCVR（目標比較付き） |
| 売上推移 | 日別売上グラフ + 累計 |
| 残日数 | ローンチ残り日数 |
| 着地予測 | 現在のペースでの最終売上予測 |

##### 3.7.3 AIボトルネック診断

数値入力ごとに自動実行:

| 出力 | 説明 |
|------|------|
| ボトルネック特定 | CVRが目標を下回るステップを特定 |
| 原因仮説 | 3つの仮説を提示 |
| 推奨アクション | 即実行できる施策リスト |
| 追加コンテンツ即生成 | 施策に対応する配信文/LP改善案をワンクリック生成 |

##### 3.7.4 良かったことの分析

| 分析 | 説明 |
|------|------|
| 高CVRステップ特定 | 想定より良い箇所の要因分析 |
| 効果的だった配信特定 | 配信後の数値変動から効果的な配信を特定 |
| 勝ちパターン抽出 | ナレッジベースに自動保存 |

##### 3.7.5 プロジェクトタイプ別ダッシュボード

| タイプ | ダッシュボード |
|--------|--------------|
| フルローンチ | 全ステップ・日次・14〜30日間 |
| スポット販売 | 簡易版・配信数/クリック/売上のみ・3〜7日間 |
| エバーグリーン | 月次サマリー・長期トレンド |

##### 3.7.6 ローンチ後レポート（PDF出力対応）

ローンチ終了時に自動生成:

| セクション | 内容 |
|-----------|------|
| 結果サマリー | 売上/成約数/ROAS |
| ファネル実績 vs 目標 | 全ステップのCVR比較表 |
| 勝因分析 | 何が売上に貢献したか |
| 改善ポイント | 次回改善すべき箇所 |
| 次回ローンチへの提言 | 具体的な改善施策リスト |
| ナレッジ自動保存項目 | 保存された勝ちパターン一覧 |

- **PDF出力**: クライアント共有用に整形済みPDF

#### 3.8 M08: エクスポート & 連携モジュール

**目的**: 生成したコンテンツを外部ツールへスムーズに連携する。

##### 3.8.1 エクスポート先

| コンテンツ | エクスポート先 | 方式 |
|-----------|--------------|------|
| LPコピー | UTAGE | 手動コピペ（セクション単位でコピーしやすいUI） |
| 配信文（LINE/メール） | UTAGE | 手動コピペ（1通ずつコピーしやすいUI） |
| SNS投稿文 | SNS各種 | テキストコピー |

## 4. 「売れる」を担保するAI生成品質設計

### 4.1 マーケティング原則の組み込み

全コンテンツ生成時にAIが自動適用する原則:

| 原則 | 適用箇所 |
|------|---------|
| 3つのNOT（読まない/信じない/行動しない）の突破 | 全LP |
| GDTの法則（Goal/Desire/Teaser） | ヘッドライン |
| OATHフォーミュラ | ペルソナ認知レベル→トーン自動調整 |
| 社会的証明の多層配置 | LP/配信/セミナー |
| 損失回避バイアス | 緊急性/限定性セクション |
| ザイオンス効果 | ステップ配信の接触設計 |
| 一貫性の原理 | ファネル全体のメッセージ一貫性チェック |
| 希少性の原理 | オファー設計 |
| 返報性の原理 | リードマグネット/無料セミナーの価値設計 |
| コントラスト効果 | 価格アンカリング |

### 4.2 修正を減らす3段階インプットシステム

```
Stage 1: プロジェクト基盤設定（初回のみ）
  ├── 商品情報の詳細入力
  ├── ペルソナ設計（AI壁打ち対話形式）
  ├── オファー設計
  ├── トンマナ設定
  └── ナレッジ登録

Stage 2: ファネル設計（ローンチごと）
  ├── ファネルテンプレ選択 or カスタム
  ├── KPI設定（売上目標→逆算）
  └── スケジュール設定

Stage 3: 各コンテンツ生成（モジュールごと）
  ├── AI壁打ち → ドラフト生成
  ├── コピースコアリング → 弱い箇所を自動改善
  ├── 円香レビュー → 対話的に修正指示
  └── 確定 → エクスポート
```

### 4.3 トーン学習

- 円香の過去の配信文/LP/SNS投稿からトーン・文体を学習
- 生成時に円香らしい表現を自動反映
- クライアント案件の場合はクライアントのトーンを別途学習
